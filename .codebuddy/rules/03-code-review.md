# 代码审查规则

## 概述
本项目采用系统化代码审查流程，覆盖安全、性能、正确性、可维护性、测试等维度。

## 审查维度与优先级

| 维度 | 关注点 | 优先级 |
|------|--------|--------|
| 安全 | 漏洞、认证、数据泄露 | Critical |
| 性能 | 速度、内存、扩展性瓶颈 | High |
| 正确性 | 逻辑错误、边界情况、数据完整性 | High |
| 可维护性 | 可读性、结构、前瞻性 | Medium |
| 测试 | 覆盖率、质量、可靠性 | Medium |
| 文档 | 注释、API 文档、变更日志 | Low |

## 严重等级

| 等级 | 标签 | 含义 | 是否阻塞 |
|------|------|------|---------|
| Critical | `[CRITICAL]` | 安全漏洞、数据丢失、生产崩溃 | 是 |
| Major | `[MAJOR]` | Bug、逻辑错误、性能显著退化 | 是 |
| Minor | `[MINOR]` | 可降低维护成本的改进 | 否 |
| Nitpick | `[NIT]` | 风格偏好、命名建议、微调 | 否 |

## 审查流程（三遍法）

### 第一遍：高层结构（2-5分钟）
1. 阅读变更描述和关联问题
2. 扫描文件列表 — 变更范围是否合理？
3. 检查整体方案 — 这是解决问题的正确方式吗？
4. 确认变更未引入架构偏移

### 第二遍：逐行审查（主要时间）
1. 从上到下阅读每个文件变更
2. 对照清单检查每个函数变更
3. 验证每个 I/O 边界的错误处理
4. 标记任何让你犹豫的地方

### 第三遍：边界与加固（5分钟）
1. 思考生产环境可能出现的问题
2. 检查已标记代码路径的测试覆盖
3. 验证回滚安全性
4. 确认文档和变更日志已更新

## 安全检查清单
- [ ] SQL 注入：所有查询使用参数化语句
- [ ] XSS：用户内容在渲染前已转义/清理
- [ ] CSRF 保护：状态变更请求需要有效 CSRF 令牌
- [ ] 认证：每个受保护端点验证用户身份
- [ ] 授权：资源访问限制在用户权限范围内
- [ ] 输入验证：所有外部输入在服务端验证类型、长度、格式
- [ ] 密钥管理：源代码中无 API 密钥、密码、令牌
- [ ] 依赖安全：新依赖来自可信来源，无已知 CVE
- [ ] 敏感数据：PII、令牌、密钥不记录到日志或错误消息中

## 反馈原则
- **具体**：指向确切行号，解释问题
- **说明原因**：陈述风险或后果，而非仅规则
- **建议修复**：提供具体替代方案或代码片段
- **提问而非命令**：主观点使用"你觉得…怎么样？"
- **认可好工作**：表扬优雅的解决方案
- **区分阻塞/非阻塞**：使用严重等级标签

## 审查反模式（避免）
- **橡皮图章**：未阅读就批准
- **自行车棚讨论**：花30分钟辩论变量名，忽略竞态条件
- **阻塞风格**：因 linter 应处理的格式拒绝批准
- **把关**：要求使用个人偏好的方案
- **范围蔓延**：请求不相关的重构

## 触发时机
- 用户请求代码审查时
- 执行 `/opsx_apply` 实施变更后
- 用户提交 PR 前的预审查
