# World Scaling & Dungeon Generation — 规格

## 能力描述

将世界地图放大至当前的 10 倍（128→1280 区块），并新增"地下城"风格的独立地图生成模式。需引入分块系统解决内存和性能瓶颈。

## 输入

- 种子 (seed number)
- 地图样式: `standard` (当前开放世界) 或 `dungeon` (地下城)
- 玩家位置 (用于分块加载)

## 输出

- 10 倍大小的可探索世界
- 地下城风格的独立地图
- 分块加载/卸载系统

## 功能要求

### Ws-1: 分块系统 (Chunk System) — 前置条件

**关键背景**：当前 `world.js` 使用单个 `Uint8Array(WORLD_WIDTH * WORLD_HEIGHT * WORLD_DEPTH)` = 128×64×128 = **1,048,576 字节**（1MB）。10 倍后 = 1280×640×1280 = **1,048,576,000 字节（1GB）**，不可行。

```
当前架构:                          目标架构:
┌─────────────────────┐           ┌────┬────┬────┬────┐
│  Uint8Array(1MB)    │           │ C0 │ C1 │ C2 │ C3 │
│                     │           ├────┼────┼────┼────┤
│  全量存内存          │           │ C4 │ C5 │ C6 │ C7 │
│  全量渲染            │           ├────┼────┼────┼────┤
│                     │           │ C8 │ C9 │C10 │C11 │
└─────────────────────┘           └────┴────┴────┴────┘
                                  Chunk = 16×64×16
                                  仅加载玩家周围 N×N 个
                                  仅渲染视锥内的 Chunk
```

- SHALL 将世界拆分为固定大小的 Chunk（推荐 16×64×16 或 32×64×32）
- 每个 Chunk SHALL 拥有独立的 `Uint8Array`
- SHALL 实现 ChunkManager：
  - 根据玩家位置加载/卸载 Chunk
  - 加载范围: 玩家周围 5×5 个 Chunk (按 16×64×16 算 = 80×64×80 可见区域)
  - 超出范围 1 圈的 Chunk 卸载释放内存

#### Scenario: 分块加载
- **GIVEN** 世界大小 1280×640×1280，Chunk 大小 16×64×16 (6400 chunks 总)
- **WHEN** 玩家在坐标 (100, 40, 100)
- **THEN** 加载的 Chunk 为坐标范围 (80-111, /, 80-111) 为中心的 5×5 网格
- **AND** 共 25 个 Chunk = 25 × 16×64×16 = 409,600 字节 ≈ 0.4MB（非 1GB）

#### Scenario: 远距离移动时的 Chunk 切换
- **WHEN** 玩家从一个 Chunk 区域移动到另一个
- **THEN** 远处的 Chunk 被卸载，新区域的 Chunk 被加载
- **AND** 加载过程中不造成画面卡顿（异步或分帧加载）

### Ws-2: 10 倍地图生成

当前世界生成使用噪声生成地形 (`world.js:239+`)，10 倍后需要适配分块系统。

- 地形生成算法 SHALL 按 Chunk 级别调用，每个 Chunk 独立生成
- 噪声函数 SHALL 使用 Chunk 的世界坐标作为输入，确保接缝处连续
- 生物群系 SHALL 在更大的尺度上分布

#### Scenario: Chunk 边界连续
- **GIVEN** Chunk (0,0) 和 Chunk (1,0) 相邻
- **WHEN** 在两个 Chunk 的边界 (x=15 vs x=16) 采样高度
- **THEN** 高度差 ≤1 个方块（视觉连续，无悬崖裂缝）

### Ws-3: 10 倍渲染适配

当前 `renderer.js` 全量构建 InstancedMesh，10 倍后必须改为 Chunk 级渲染。

- SHALL 为每个加载的 Chunk 构建独立的 InstancedMesh 组
- SHALL 实现视锥体裁剪 (`camera.frustum` 检测)
- SHALL 增加 LOD (Level of Detail) 系统：
  - 近处 (0-80): 全细节方块渲染
  - 中处 (80-200): 合并为 2×2 大方块
  - 远处 (200+): 不渲染或使用 Billboard 贴图
- 雾 SHALL 适配: `fog.far = 200` (当前 180)

#### Scenario: 远距离 LOD
- **GIVEN** 玩家在位置 (0, y, 0)
- **WHEN** 距离 150 单位的方块区域被渲染
- **THEN** 远处的方块以 2×2 合并后的形式显示
- **AND** 无明显"弹出"感

### Ws-4: 地下城地图生成

新增 `style: 'dungeon'` 地图生成模式，参考《我的世界地下城》。

```
典型地下城布局:
┌───┬───┬───┬───┬───┐
│ S │   │   │ E │   │  S=出生房, E=出口
├───┼───┼───┼───┼───┤
│   │   │   │   │   │  走廊: 2格宽, 3格高
├───┼───┼───┼───┼───┤
│   │ B │   │   │   │  B=战斗房(刷怪)
├───┼───┼───┼───┼───┤
│   │   │   │   │ T │  T=宝藏房
├───┼───┼───┼───┼───┤
│   │   │   │   │   │  墙壁: 石头/砖块
└───┴───┴───┴───┴───┘  地面: 石砖/苔石
```

- SHALL 使用 BSP (Binary Space Partition) 算法生成房间连通图
- SHALL 生成以下房间类型:
  - 出生房 (起点，带火炬)
  - 走廊 (2 格宽，连接房间)
  - 战斗房 (20-50m²，生成敌对生物)
  - 宝藏房 (含宝箱)
  - BOSS 房 (终点，最终战斗)
- SHALL 室内光照系统:
  - 封闭空间无日光
  - 火炬/发光石作为唯一光源 (`PointLight`)
  - 暗处视野受限 (缩小 `fog.far` 到 30)

#### Scenario: 地下城生成
- **GIVEN** 玩家选择地下城模式
- **WHEN** 地图生成
- **THEN** 生成 6-10 个房间通过走廊连接
- **AND** 墙壁为石头材质，地面为石砖
- **AND** 入口房间有火炬光源
- **AND** 至少有一个 BOSS 房间

## 约束

- Ws-1 (分块系统) 是 Ws-2 和 Ws-3 的前置依赖
- 地下城世界 (Ws-4) 不需要分块系统（单次生成，地图规模小）
- 保持向下兼容：现有存档格式需要迁移或重建
- 分块系统不影响现有的 `/tp`、`/fly` 等命令

## 依赖

- `world.js` (generate, blocks, World class)
- `renderer.js` (buildMeshes, InstancedMesh, fog)
- `input.js` (射线检测范围)
- `camera.js` (上帝视角高度、Fog 参数)
- `main.js` (存档序列化)
- Three.js (Frustum, InstancedMesh, PointLight)
